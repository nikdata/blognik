As many data scientists who are actively looking for a new job may know, one of the more exciting (or dreaded) parts of the interview process is the coding assessment. For those that may not know, the coding assessment can vary widely from one company to another. Questions can be simple as writing a simple SQL script or as complex as writing abstract code to ‘productionize’ a machine learning model. Preparation can be daunting and if you’re like me, you might feel lost as to where to even begin! But fear not, there are lots of resources to help you prepare for coding interviews such as HackerRank or Leet Code.

One of the companies that interviewed me, asked a coding question that was - well - dumb.

Sort the following array in ascending order:

[(5, 2, 1), (3, 2), (8, 3, 5, 1)]

Possible answers could be:

  • [(2, 1, 0), (1, 0), (3, 1, 2, 0)]: this refers to the ‘index’ of the value in array

The issue I have with this kind of question is that it does not accurately nor effectively gauge a person’s (especially a Data Scientist’s) ability to code in Python, R, Julia, etc. Sorting arrays is not what I would consider a core competency for a data scientist. I know many data scientists, myself included, that are more familiar with the concept of data frames instead of arrays.

Arrays are powerful and can be efficient, but my personal position is that code efficiency is not always a priority and doesn’t have to be. Not every code is going to go to production nor does it need to scale. Rather, I’ve always been an advocate that code readability and ease of understanding is paramount - especially if the code will be read by others. I also believe that it’s a mistake if it is to be assumed that the other person reading the code is as well versed in coding as the original author.
